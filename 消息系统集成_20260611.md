## 关键修改

### 1. 后端：server.js
**位置：** 第 1526-1720 行（公告API之后）

**添加内容：**
- `GET /api/notifications` - 获取当前用户的通知（公告 + 个人通知）
- `PUT /api/notifications/:id/read` - 标记通知已读
- `POST /api/notifications/batch-read` - 批量标记已读
- `POST /api/notifications/batch-delete` - 批量删除通知

**实现逻辑：**
1. 动态将已发布的公告转换为通知

4. 按时间倒序排序（置顶公告排最前）

**数据结构：**
```javascript
// data.json 中新增
{
  "notifications": [],        // 个人通知
  "notification_reads": []    // 已读记录 { userId, noticeId, readAt }
}
```

## 技术决策

### 为什么动态转换公告而不是存储副本？
1. **节省存储空间**：不需要为每个用户存储公告副本
2. **实时性**：公告更新后，用户立即看到最新版本
3. **简化逻辑**：不需要同步更新所有用户的通知

### 为什么使用 notification_reads 记录已读状态？
1. **多设备同步**：已读状态存储在服务端，切换设备后仍然有效
2. **灵活性**：可以为每个用户独立记录已读状态
3. **可扩展**：未来可以支持"全部已读"、"按公告已读"等功能

## 附录：API 文档

### GET /api/notifications
获取当前用户的通知列表。

**Headers:**
```
Authorization: Bearer <token>
```

**Response:**
```json
{
  "success": true,
  "data": [
    {
      "id": "notice_1234567890",
      "originalId": 1234567890,
      "title": "平台公告",
      "content": "<p>公告内容...</p>",
      "type": "announcement",
      "isHtml": true,
      "read": false,
      "readAt": null,
      "createdAt": "2026-06-11T10:00:00.000Z",
      "pinned": true
    }
  ]
}
```

### PUT /api/notifications/:id/read
标记单条通知已读。

**Headers:**
```
Authorization: Bearer <token>
```

**Response:**
```json
{
  "success": true
}
```

### POST /api/notifications/batch-read
批量标记已读。

**Headers:**
```
Authorization: Bearer <token>
Content-Type: application/json
```

**Body:**
```json
{
  "ids": ["notice_123", "notification_456"]
}
```

**Response:**
```json
{
  "success": true,
  "count": 2
}
```

### POST /api/notifications/batch-delete
批量删除个人通知（不能删除公告）。

**Headers:**
```
Authorization: Bearer <token>
Content-Type: application/json
```

**Body:**
```json
{
  "ids": ["notification_456", "notification_789"]
}
```

**Response:**
```json
{
  "success": true,
  "count": 2
}
```
